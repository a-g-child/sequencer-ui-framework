# Audio Graph Engine

## 1. Goal

Sequencer's future audio system should behave like a device graph engine in the
Max/MSP or Max for Live family:

```text
MIDI / Audio / Control sources
  -> nodes
  -> device graph
  -> audio / MIDI / control outputs
```

The existing Sequencer contract should stay intact:

```text
UI edits intent.
Document stores graph.
Runtime builds executable graph.
Native engine executes graph.
```

The UI framework should edit the graph document model and inspect the resolved
runtime graph. It should not run DSP or own audio scheduling.

## 2. Relationship To Current RuntimeDevice Architecture

The current `DeviceDescriptor`, `DeviceInstance`, `RuntimeDevice`, and playback
event path remain the creative-facing contract while the graph engine is
introduced.

Graph presets on device descriptors should be treated as reusable graph
fragments, not as the final execution owner. A device can describe its internal
fragment, but the device chain assembles the executable track graph:

```text
Track
  -> Device Chain
  -> Track Graph
  -> Executor
```

This keeps linear chains, modulation routing, sends, parallel paths, and
sidechains in one schedulable graph instead of scattering ownership across
individual devices.

Today, instruments such as Basic Synth and Sampler are hardcoded runtime device
classes. The graph engine moves those instruments toward graph presets assembled
from reusable nodes:

```text
Basic Synth Device
  MIDI source
  -> oscillator
  -> filter
  -> ADSR gain
  -> track mixer
  -> output
```

```text
Sampler Device
  MIDI source
  -> sample player
  -> filter
  -> ADSR gain
  -> track mixer
  -> output
```

During migration, a `RuntimeDevice` can continue to interpret notes, voices, and
parameters while exposing or consuming a graph document. Later, graph-backed
devices can produce commands for WebAudio or native executors without changing
the document or UI editing model.

## 3. Graph Document Model

The document model is serializable creative intent:

- `AudioGraphDocument`: graph id, version, nodes, connections, and optional
  metadata.
- `AudioGraphNode`: node id, descriptor type, display name, parameter values,
  and optional position data for editors.
- `AudioGraphConnection`: source node/port to target node/port.
- `AudioNodeDescriptor`: reusable node vocabulary with typed ports, parameter
  definitions, category, latency hints, and execution metadata.

The document model should avoid executable objects, class instances, WebAudio
nodes, native handles, mutable registries, and UI state.

## 4. Runtime Graph Model

The runtime graph model is derived from a document and descriptors:

- resolved nodes with their descriptors
- resolved typed ports
- validated connections
- execution order
- diagnostics
- latency hints

The builder should fail softly with diagnostics where possible so the UI can
show incomplete or invalid graphs while the user is editing.

## 5. Node Categories

Initial node categories:

- Sources: MIDI input, clip note source, audio file player, oscillator, noise.
- Processors: ADSR / volume envelope, filter, gain, pan, mixer, delay,
  compressor.
- Control / modulation: LFO, envelope follower, macro knob, automation input,
  random/sample and hold.
- Outputs: audio out, MIDI out, device/module out.

The first TypeScript package should define descriptors for the node vocabulary
without implementing DSP.

## 6. MIDI, Audio, And Control Port Types

Ports must be typed before runtime execution:

- `midi`: note, controller, clock, and device MIDI messages.
- `audio`: sample streams with a channel count.
- `control`: scalar or signal-rate parameter/control values.

Connections are valid only when the source and target port types match. Audio
ports should additionally validate channel compatibility when a target declares
a fixed channel count.

## 7. Parameter Automation And Modulation

Parameters remain saved values in the document. Automation and modulation should
arrive as graph inputs rather than hidden side channels:

```text
automation input -> parameter target
LFO -> parameter target
macro knob -> multiple parameter targets
```

The runtime graph should distinguish:

- base parameter value saved in the document
- automation value scheduled from the playback model
- modulation value generated by graph nodes
- effective value used by the execution backend

The UI can edit base values and routing. Runtime backends resolve effective
values.

## 8. Native Execution Boundary

Native execution sits below the runtime graph:

```text
AudioGraphDocument
  -> AudioGraphBuilder
  -> RuntimeAudioGraph
  -> WebAudio / native adapter
  -> DSP engine / driver
```

Native code owns:

- real-time DSP
- buffers and channel layouts
- sample-accurate scheduling
- audio thread command queues
- driver integration
- metering and latency reporting

Native code should not own:

- saved project truth
- UI state
- graph editing operations
- device browser metadata
- Svelte objects or TypeScript class instances

## 9. WebAudio As Reference Backend

WebAudio should be the first executable backend because it is debuggable,
browser-native, and close to the current app. It should be treated as a
reference backend, not the long-term execution contract.

The WebAudio backend can prove:

- graph descriptor coverage
- parameter mapping
- connection validation
- voice and note command translation
- parity with current Basic Synth and Sampler behavior

Native engines should later match the same graph/runtime contract.

## 10. Migration Path From Basic Synth And Sampler

The first implementation slice is TypeScript-only:

```text
packages/audio-graph
  AudioGraphDocument
  AudioNodeDescriptor
  AudioGraphNode
  AudioGraphConnection
  AudioGraphBuilder
```

Milestones:

1. Define the graph document and descriptor vocabulary.
2. Express Basic Synth as a graph preset.
3. Add builder diagnostics and execution order resolution.
4. Add Sampler as a graph preset.
5. Add a WebAudio reference executor behind the runtime graph.
6. Route current RuntimeDevice voice actions into graph executor commands.
7. Move Basic Synth and Sampler runtime behavior to graph-backed execution.
8. Add a native adapter that consumes the same runtime graph and command stream.

Current bridge milestone:

```text
Device graph presets
  -> buildDeviceChainGraph(...)
  -> track-level RuntimeAudioGraph diagnostics
```

This composes MIDI effects, instruments, and audio effects into one track graph
while the existing runtime device path continues to execute sound generation.

The current hardcoded devices can remain in place until graph-backed devices
match their behavior.
